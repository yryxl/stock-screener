# 📋 系统需求变更记录（REQUIREMENTS）

> **这是本项目从诞生至今的所有功能决策和修改的完整记录。**

**文档目的**：
1. 避免改来改去改回原样
2. 换 AI / 换电脑 / 换账号时，能快速理解"来时的路径"
3. 每次修改必须在此登记，作为"契约文档"

**维护规则**：
- ✅ **新增需求前必须读本文档**，避免重复/冲突/回退
- ✅ **实施修改后立即在本文档更新**
- ✅ **每条需求必须包含**：ID / 时间 / 提出原因 / 修改内容 / 涉及文件 / 验收标准 / Commit
- ❌ **不要删除旧条目**，即使废弃也用「❌ 撤销」标记并附原因
- ⏱️ 需求按**功能模块**分类，模块内按**时间线**排列

## 🚨 强制惯例（用户 2026-04-18 明确，与 TESTING.md BUG 表同等约束）

**凡需求必入表**，**任何**类别都不例外：
1. 用户主动提的需求
2. 调研发现的遗漏（如巴菲特/芒格理念盘点）
3. 设计阶段的子需求（拆分大需求时）
4. 优化重构（即使不改外部行为）
5. 文档/规范类（也要立项）
6. 数据维护类（如更新 cape_data.json 等）

**🔒 不可破坏的规则**（与 BUG 表完全一致）：
1. **只叠加不覆盖**：REQ 表里的旧记录**绝对不能删除/改写/合并**
   - 即使被新版本替代（如 REQ-186 v1 → v2），也是**追加 v2 章节**，v1 内容保留
   - 状态更新只能改"实施状态"字段，不能"删除整章"
2. **必须记录实施方法**（每条 REQ 都要写）：
   - 不能只写"已实施"，要写**用什么方案实施的**
   - 例：✅ 已实施 — 方案 A（禁用方案 B 因为成本太高），见 [代码定位]
   - 7 要素模板的"4. 是否有其它方案 + 选择理由"和"6. 代码定位"是必填
3. **REQ ID 顺延不复用**：REQ-001~191、REQ-FIX-001/002、REQ-VERIFY-001 已用过
   - 下次新增按编号往后顺延，**绝不复用旧编号**
   - 子需求用小数点（如 REQ-186.v2 / REQ-185 维度 4）
4. **撤销也要保留**：如果某需求废弃，加 ❌ 撤销标记 + 原因，**不删除原文**

**记录时机**：
- 用户提需求立刻在表里加一行（不要等实施完才加）
- 实施完成后更新"实施状态"+"代码定位"+"Commit"字段
- 后续发现需要扩展，**追加新章节**而不是改旧的

**目的**：
- 避免改来改去改回原样（用户原话）
- 接手 AI 能看到完整决策链路 + 学到当时的方案选择
- 复盘时能定位"哪类需求最多被反复改"，优化设计流程
- 同一需求多次重写 → 暴露需求理解问题，应升级沟通而非每次重做

**当前版本**：v1.5（2026-04-18 调研巴菲特/芒格"模型抓不到"理念 + 浑水方法论 +
A 股造假识别，新增 4 项 P1 工作：036/037/038/039。news_screen skill 扩展为 6 层

## 📐 需求记录模板规范（v1.2 新增）

**问题背景**：2026-04-17 发现 AI 接手时容易把不同概念混淆（典型案例：把"市场极冷策略响应"误当成"模型可靠性熔断"，TODO-001 因此走偏一次）。

**解决方案**：每条新需求必须包含 7 要素，让接手者一看就明白"这是干什么的、做到哪、怎么验"。

### 7 要素模板

```
### REQ-XXX：[需求标题]（[日期] [状态：✅完成/⏳进行中/📋未启动]）

#### 1. 应对什么情况
[触发场景/背景。说清楚是什么具体问题让我们要做这件事]

#### 2. 解决什么问题
[核心痛点。如果不做会发生什么具体的坏事]

#### 3. 最终效果（用户能看到什么）
[用户视角的可见结果。比如"前端在 X 时显示 Y banner，文案 Z"]

#### 4. 是否有其它方案 + 选择理由
[列出备选方案（至少 2 个），说清楚为什么选这个]

#### 5. 实施状态
[未启动 / 进行中 X% / 已完成。当前到哪一步、还差什么]

#### 6. 代码定位
[文件路径 + 关键行号。直接能跳到代码]

#### 7. 验收标准
[怎么算做对了。可执行的 yes/no 检查项]
```

### 何时使用

- ✅ **新增需求必须按此模板**
- ✅ **关键历史需求逐步迁移**（优先：REQ-151/152/153/160-191、所有 TODO 项）
- ⚠ **小修小补可以简化**（比如 typo 修复、一行文案改动）

### 防 AI 理解偏差的关键点

- **第 1 要素"应对什么情况"必须写具体场景**：不能写"提高模型质量"这种模糊描述，要写"用户反馈：xxx"或"测试发现：xxx"
- **第 4 要素"是否有其它方案"必须列至少 2 个**：避免接手者重复探索已经否决的方案
- **第 6 要素"代码定位"必须含文件名+行号**：没有定位就找不到上下文

## ⚠️ 已知局限（系统性风险）

**必读**：本节记录模型的两大根本性假设风险。使用模型者必须先了解这些局限，再决定如何使用。

### 局限 1：查、巴理念是否适用中国股市？

**担心**：查、巴理念来自美股 100+ 年历史，中国股市仅 30 年。生态不同，生硬照搬可能水土不服。

**已核查**：**部分适用，部分不适用**。

#### ✅ 完全适用的部分（不分国别）
- ROE 是核心指标（数学规律）
- 有护城河的公司值得长期持有（经济学规律）
- 别为一个故事付高价（估值数学）
- 宁可错过不犯错（人性规律）

#### ⚠️ 需要调整的中国特色风险
1. **政策风险** 远超美国
   - 教培 2021、互联网反垄断 2021、地产三条红线 2021
   - 一纸文件可灭整个行业，模型看不到政策信号
2. **"十年 ROE ≥ 15%" 标准过严**
   - A 股龙头公司上市时长普遍短于美股
   - 容易把真正优秀的年轻公司错分为"平庸"
3. **分红文化弱**
   - A 股很多公司不分红或少分红
   - "股息率打分"对 A 股系统性不利
4. **行业代际变化快**
   - 家电 → 互联网 → 新能源 → AI
   - 过去的"十年王者"不一定是未来的王者
5. **散户主导**
   - A 股散户占比 60%+，羊群效应强
   - 牛更疯、熊更狠，"85% 分位泡沫" 可能更频繁

#### 🤔 参考事实
- **巴菲特在中国只重仓过比亚迪一只**（2008-2022）
- **芒格买过阿里，亏了 40%**
- 巴菲特本人对中国市场非常谨慎

### 局限 2：历史数据的局限性

**担心**：出现历史中没出现过的超长熊市或牛市，模型可能失效。

**已核查**：**模型有 4 个脆弱点**。

#### 🧊 脆弱点 1：PE 分位是"均值回归"思维
- 假设 PE 85% 分位会回到中位数
- **但结构性转变时 85% 可能成为新常态**
- 日本 1989-2009："20 年失去的时光"，PE 从 60 倒 15 后继续横盘 20 年

#### 🧊 脆弱点 2：ROE 稳定性假设
- 用历史 ROE 预测未来
- AI 革命可能颠覆传统行业的 ROE 基础
- 教育、医疗、咨询行业可能被 AI 打掉一半毛利

#### 🧊 脆弱点 3：回测过拟合
- 规则用 2001-2025 数据调优
- 过去 24 年是 A 股崛起+城镇化+WTO 红利
- 下一个 24 年是老龄化+去全球化+产业外移
- **宏观环境完全不同，规则可能失效**

#### 🧊 脆弱点 4：黑天鹅免疫缺失
- 2020 新冠、2008 危机、2011 福岛
- 历史数据无法覆盖前所未有的事件

## 一、核心选股模型

### REQ-001：初始筛选系统（芒格体系 + PE 分级买卖信号）
- **时间**：项目初期
- **原因**：需要一个基于巴菲特/芒格价值投资的基础选股框架
- **内容**：
  - 基础财务检查（ROE、债务、毛利率、现金流）
  - 按 PE 给出买入/持有/卖出信号
- **文件**：`screener.py`
- **Commit**：`1365f7f`

### REQ-002：模型重构为"清单筛选 + 打分排序 + 股息率"
- **原因**：单纯 PE 信号不够，需要多维度评估
- **内容**：
  - 清单筛选决定买不买（1 vote）
  - 多维度打分决定排序（ROE质量、财务健康、盈利能力、股息率、估值）
  - 总分 0-50 分制
- **文件**：`screener.py`, `scorer.py`
- **Commit**：`e13da39`

### REQ-003：ROE 标准对齐巴菲特 1987 年股东信
- **原因**：默认 ROE ≥ 15% 对不同行业不合理
- **内容**：按复杂度分档
  - 简单生意：ROE ≥ 15%
  - 中等生意：ROE ≥ 17%
  - 复杂生意：ROE ≥ 20%
- **文件**：`screener.py`（`COMPLEXITY_ROE_ADJUST`）
- **Commit**：`8acb112`

### REQ-004：合理价格买好公司（巴菲特 1989 年股东信）
- **原因**：好公司即使在合理区间也应该考虑买入，不是非要等到便宜
- **内容**：
  - 好公司（10 年 ROE ≥ 15% + 毛利 ≥ 30%）+ PE 在合理区间内 → `buy_light` 或 `buy_watch`
  - 不是好公司即使 PE 便宜也不推荐加仓
- **文件**：`screener.py`, `backtest_engine.py`
- **Commit**：`bdef22e`

### REQ-005：十年王者规则（核心理念：熊市买十年王者）
- **原因**：巴菲特持仓的可口可乐、GEICO 都是 10 年连续 ROE ≥ 15% 的公司
- **内容**：
  - 定义：连续 10 年 ROE ≥ 15% + 最低年 ROE ≥ 12%
  - 十年王者即使偏贵也优先持有，熊市来时重点加仓
- **文件**：`live_rules.py`（`check_10_year_king_live`）
- **Commit**：`6cec5af`

### REQ-006：好公司放宽为"5 年 ROE 均值 ≥ 20% + 毛利 ≥ 30%"
- **原因**：十年门槛太严，错过许多优质但上市不满 10 年的公司
- **内容**：新增 `is_good_quality_live` 判定（次于十年王者）
- **文件**：`live_rules.py`
- **Commit**：随 REQ-005 合并

## 二、信号分级体系

### REQ-010：完善信号体系（买入 4 级 + 持仓 1 级 + 卖出 5 级）
- **原因**：初版只有 buy/sell/hold 三档，不够细
- **内容**：
  - 买入：`buy_heavy` / `buy_medium` / `buy_light` / `buy_watch`
  - 持有：`hold_keep`（持仓专用）、`buy_add`（持仓可加仓）
  - 卖出：`sell_watch` / `sell_light` / `sell_medium` / `sell_heavy` / `true_decline`
- **文件**：`screener.py`
- **Commit**：`9c6a5ab`

### REQ-011：关注表 PE ≥ 合理区间一律"观望"
- **原因**：关注表是"建仓前观察"阶段，不该给卖出信号
- **内容**：即使股票显示 sell 信号，在关注表里一律显示"hold"观望文案
- **文件**：`main.py`（`check_watchlist`）
- **Commit**：`7e6f5dd`

### REQ-012：加仓信号仅限十年王者/好公司
- **原因**：发现云南白药这种 10 年 ROE 均值 13%（未达 15%）的平庸企业也被建议加仓，违反"别给平庸企业摊平"的芒格原则
- **内容**：
  - 持仓股有买入信号时：
    - 十年王者 → `buy_add`
    - 好公司 → `buy_add`
    - **平庸企业** → 降级为 `hold_keep`（持有不加仓）
- **文件**：`screener.py`
- **Commit**：`434269d`

### REQ-013：screener 去除 "持有不加仓 / 可轻仓买入" 矛盾文案
- **原因**：用户收到消息 `"非十年王者/好公司 → 建议持有不加仓 | PE=18.9低于行业合理区间20-30→可以轻仓买入"`，同一文案既说"不加仓"又说"可轻仓买入"，自相矛盾
- **内容**：降级为 `hold_keep` 时**不再拼接**原"可轻仓买入"文案，改为清晰单一说明
  - 改前：`"非十年王者/好公司 → 建议持有不加仓 | {原PE文案}"`
  - 改后：`"非十年王者/好公司（ROE未达15%连续10年标准）→ 虽然PE偏低但不加仓，等ROE真正恢复到15%+再考虑"`
- **文件**：`screener.py`
- **Commit**：**本次新增（待推送）**
- **验收**：新文案中不同时出现"不加仓"和"买入"两个动作词

## 三、行业与复杂度分类

### REQ-020：区分简单/中等/复杂生意
- **原因**：芯片半导体等复杂行业 PE 天然比白酒高，统一标准会漏掉
- **内容**：
  - 简单生意（白酒/食品/银行等）：PE 8-30
  - 中等生意（医疗器械/电力等）：PE 12-35
  - 复杂生意（半导体/军工/新能源等）：PE 25-65
- **文件**：`screener.py`（`INDUSTRY_PE`）
- **Commit**：`6606eb2`

### REQ-021：周期股彻底过滤
- **原因**：周期股（钢铁/煤炭/有色等）PE 失真，ROE 波动大，不符合巴菲特"简单生意"标准
- **内容**：
  - `CYCLE_STOCKS_ENABLED = False`（默认关闭周期股推荐）
  - 模型不给周期股买卖信号
- **文件**：`backtest_engine.py`
- **Commit**：`b6904b5`

### REQ-022：行业自动获取（stock_zh_a_spot_em 不返回行业）
- **原因**：默认数据源不含行业字段，导致所有股票走 `DEFAULT_PE {15-25}` 兜底，误判严重
- **内容**：
  - 新增 `get_stock_industry(code)`：多层 fallback
    1. `stock_individual_info_em` 接口
    2. 本地 `stock_industry_cache.json` 缓存
    3. 持仓/关注表 `category` 字段兜底
  - 缓存避免重复调接口
- **文件**：`data_fetcher.py`, `stock_industry_cache.json`
- **Commit**：`46b6cc2`, `7183af3`

## 四、护城河与基本面检查

### REQ-030：真假下跌判断
- **原因**：股价下跌不一定是公司恶化，可能是市场情绪
- **内容**：
  - 检查 ROE 连续下滑 + 现金流 + 毛利率
  - 真跌 → `true_decline`（基本面恶化）
  - 假跌 → 按 PE 给卖出信号
- **文件**：`screener.py`
- **Commit**：`6e076ab`

### REQ-031：8 条护城河规则完整检查
- **原因**：基础财务检查不够，需要更严格的护城河判断
- **内容**：
  - 规则 1-7：ROE < 0 / ROE 大幅下滑 / ROE 连续下滑 / 毛利连续下滑 / 营收下滑+ROE低 / 债务飙升+ROE下滑 / 现金流比率 < 0.3
  - **规则 8**：5 年 ROE 单调下降且累计跌幅 ≥ 10pp 且最新 < 20%（"温水煮青蛙"检测）
- **文件**：`live_rules.py`（`check_moat_live`）, `backtest_engine.py`（`check_moat_normal`）
- **Commit**：`0de8714`

### REQ-032：ROE-PB 背离检查（规则 8 补充）
- **原因**：ROE 下降但 PB 没跌 = 估值假象（市场没反映基本面恶化）
- **内容**：
  - ROE 下降 ≥ 5pp + PB > 1.5 → 降级为 hold
  - 用真实 PB（`stock_zh_a_spot_em` "市净率"字段）
- **文件**：`screener.py`（`check_fundamental_health`）, `backtest_engine.py`
- **Commit**：`4a56ad2`, `713e76a`

### REQ-033：消费龙头现金流警示（豁免但重点关注）
- **原因**：某些消费龙头（茅台、伊利）ROE 和毛利很高但现金流比率波动，机械按规则 7 会误杀
- **内容**：
  - 高 ROE + 高毛利的消费龙头豁免规则 7

### 📊 好公司分级体系（统一 4 个既有函数成 3 档分级）

### 📚 概念关系表（芒格定性概念 ⊂ 原模型静态标签）

**关键：⊂ 是子集关系，不是等价！**两套术语属不同维度：
- 原模型 `complexity/type` 是**静态行业分类**（基于行业特征）
- 芒格术语是**实证表现**（基于 ROE/资本开支等财务数据）
- 同一行业内既可能有跑步机也可能有"非跑步机"（半导体里有台积电也有京东方）

| 芒格术语 | 原模型标签 | 关系说明 | 典型案例 |
|---|---|---|---|
| **简单生意** | complexity=simple | 两者基本重合 | 白酒、消费、家电、中药 |
| **跑步机型** | ⊂ complexity=complex | 复杂生意 **+ ROE 长期偏低** 的那部分 | 钢铁、面板、航空（但不含 ROE 高的半导体如台积电） |
| **冲浪者型** | ⊂ complexity=complex 或 type=cycle（技术类） | 必须持续技术迭代才能活（煤炭虽是 cycle 但**不是**冲浪者） | 半导体、光伏、锂电设备、消费电子 |
| **过路费生意** | ⊂ type=utility | 公用事业 **+ 特许经营稳定** 的那部分（光伏发电虽是 utility 但**不是**过路费——可替代） | 铁路、港口、水电、高速、燃气、电信 |
| **下水道生意** | ⊂ complexity=complex 或 type=cycle | 10 年 ROE 实证弱（均值<8% 或频繁<5% 或频繁亏损） | 中铝、京东方、中国石油、宝钢、三大航空、鞍钢 |

**芒格"3 类不能买"**：下水道（资本黑洞）、跑步机（重资产焦虑）、冲浪者（技术迭代焦虑）
**芒格"3 类好生意"**：垄断、过路费、简单的加价（消费品提价）

造假股不是独立一类，是"下水道/跑步机/冲浪者"的道德败坏形式—— REQ-160 已对齐。

### REQ-VERIFY-001：21 项语录建议上网校验结果（2026-04-16）

**背景**：REQ-CONCEPT-002 推出 21 项建议后，用户要求上网校验是否确实出自巴菲特/芒格原话，阈值是否合理，是否适用中国国情。Agent 用 WebSearch 对高优先级 8 项 + 关键中优先级 3 项做了深度校验。

#### 🔴 必须撤回（1 项）
- **REQ-188 追踪止损 20%**：巴菲特**明确反对止损**（原话："像喜欢的房子有人出价低 10% 就卖"）。20% 这个数字不来自任何芒格语录，属于技术派思路。A 股单月 20% 回撤常见，规则与"基本面未变就持有"核心理念冲突。

#### ⚠ 概念/阈值有误，需大幅调整（4 项）
- **REQ-184 "10% CAGR 倒推"**：巴菲特原话的"10%"是 **required return（要求回报率）**，不是 CAGR 增速假设。折现率他用**长期国债收益率**，不是 10%。
  → 正确做法：假设 `10 年持有期要求 10% 年化回报`，结合保守 EPS 折现倒推。
- **REQ-187 "极冷加仓上限 90%"**：伯克希尔历史现金比例 **2008Q2 约 15-20%、2020/3 约 36%、2024Q3 约 28%**——**从未把股票仓位推到 90%**。
  → 正确做法：极冷温度提示加仓可保留，但**总仓位上限改为 80%**（对齐伯克希尔实操）。
- **REQ-182 "利率 4% 触发 PE 收窄"**：4% 对应美国历史，**中国 10 年国债 2015-2024 区间 2.5-4.0%，当前 1.8% 在历史低位，4% 已是历史高位**。
  → 正确做法：触发线改 3.0-3.5%。
- **REQ-179 "业绩标准差 <3%"**：麦道夫是"月度"回报稳定，不是年度；**A 股造假特征是存贷双高、存货异常**（已在 REQ-160），不是平滑。
  → 正确做法：多条件（CV<5% + OCF/净利润 <0.8 + 连续 7 年）才触发，作为辅助信号。

#### 🟢 原则+阈值扎实，可直接接入（2 项）

## 🐛 REQ-FIX-001：data_fetcher 字段名 bug 修复（2026-04-17 ✅ 已修）

**背景**：Phase 3 端到端测试发现茅台 / 全部 15 只股票均未通过模型，根因是 `check_opm_stable` 与 `check_fcf` 都报"数据不足"。

**根因**：
- akshare `stock_financial_analysis_indicator` 实际返回的字段名是：
  - 利润率：**销售净利率**（不是"营业利润率"或"销售利润率"）
  - 现金流：**每股经营现金流**（不是"每股经营性现金流"或"每股经营现金流量"）
- `data_fetcher.find_column` 是**子串匹配，不支持正则**
- 老代码以为 `经营.*现金` 是正则 fallback，实际无效
- 导致 `find_column(["营业利润率", "销售利润率"])` 永远返回 None

**修复**（[data_fetcher.py:307](../data_fetcher.py#L307)）：
- `get_opm_series`：候选列表加 `销售净利率`
- `get_fcf_series`：候选列表加 `每股经营现金流`，fallback 改为 `经营现金 / 经营性现金` 子串

**验收**：
- ✅ 茅台 OPM = 51.1%（销售净利率）
- ✅ 茅台 FCF 充足
- ✅ 全部 17 只股票字段填充完整，0 崩溃

## 🐛 REQ-FIX-002：印钞机 / 管理层评分 None 边界保护（2026-04-17 ✅ 已修）

**背景**：第五阶段边界测试发现 2 个 None 处理隐患。

### 问题 1：印钞机标签传 None ROE 时被错误标为"卓越印钞机"
- **根因**：`check_cashcow_label` 的 ROE 门槛检查写法
  ```python
  if roe_5y_avg is not None and roe_5y_avg < 20:
      return None
  ```
  当 `roe_5y_avg = None` 时条件不成立，**绕过了门槛检查**，继续往下走，最后只看 CapEx/净利就给标签
- **修复**（[china_adjustments.py:1631](../china_adjustments.py#L1631)）：
  ```python
  if roe_5y_avg is None:
      return None, "", ""  # 数据缺失直接拒绝判定
  if roe_5y_avg < 20:
      return None, "", ""
  ```
- **验收**：传 ROE=None 时返回 `(None, "", "")` ✅

### 问题 2：管理层评分 3 个维度全空仍返回 100 分
- **根因**：`check_management_scorecard` 从 score=100 起步只做减分。如果 3 个维度（质押/商誉/分红）全部拿不到数据，没有任何减分发生，最终输出 100 分（被错误标为"优秀"）
- **修复**（[china_adjustments.py:1018](../china_adjustments.py#L1018)）：引入 `dimensions_evaluated` 计数器，当 3 维度全为 0 时返回 `score=None, tier="数据不足"`
- **调用方适配**（[screener.py:578](../screener.py#L578)）：检查 `mgmt["score"] is not None` 后才参与分档判断
- **验收**：未在生产环境触发（前置 `get_financial_indicator` 已拦截无效代码），仅作为加固保护
